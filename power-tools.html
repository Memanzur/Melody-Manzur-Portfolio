gstack by Garry Tan

The Sprint Workflow

gstack is a process, not a collection of tools. The skills run in the order a sprint runs. Each skill feeds into the next. Nothing falls through the cracks because every step knows what came before it.

1

Think

Start with /office-hours. Six forcing questions reframe your product before you write code. Pushes back on your framing, challenges premises, generates implementation alternatives.

2

Plan

CEO review rethinks the problem. Eng review locks architecture. Design review rates every dimension 0-10. The plan gets challenged from every angle before a line of code is written.

3

Build

Write the code with AI. gstack's continuous checkpoint mode auto-commits your work as you go. Survives crashes and context switches.

4

Review

/review finds bugs that pass CI but blow up in production. Auto-fixes the obvious ones. /codex gets a second opinion from a completely different AI model.

5

Test

/qa opens a real browser, clicks through flows, finds bugs, fixes them with atomic commits, and generates regression tests. The agent has eyes now.

6

Ship

Sync main, run tests, audit coverage, push, open PR. /land-and-deploy merges, waits for CI, and verifies production health.

7

Reflect

/retro runs a team-aware weekly retrospective. Per-person breakdowns, shipping streaks, test health trends, and growth opportunities.

Superpowers by Jesse Vincent

Superpowers is an agentic skills framework and software development methodology created by Jesse Vincent at Prime Radiant. It works across Claude Code, OpenAI Codex, Cursor, Gemini CLI, and GitHub Copilot. Instead of letting your coding agent jump straight into writing code, Superpowers forces a real process: spec first, plan second, build third.

The core idea: Your coding agent does not just write code. It steps back, asks what you are really trying to do, teases a spec out of the conversation, then shows it to you in chunks short enough to actually read. After you sign off on the design, it creates an implementation plan clear enough for an enthusiastic junior engineer to follow. Then it launches subagents to work through each task, reviewing their work as they go.

Superpowers enforces true red/green test-driven development. Write a failing test, watch it fail, write minimal code to make it pass, commit. No code gets written before a test exists. Combined with YAGNI (You Aren't Gonna Need It) and DRY principles, it keeps agents from overbuilding.

The result: Claude can work autonomously for a couple hours at a time without deviating from the plan you put together. The skills trigger automatically, so you do not need to do anything special. Your coding agent just has Superpowers.

"Because the skills trigger automatically, you don't need to do anything special. Your coding agent just has Superpowers."
Jesse Vincent, creator of Superpowers

The Superpowers Workflow

1

Brainstorm

Before any code, the agent refines your rough idea through questions. Explores alternatives, presents the design in digestible sections for you to validate. Saves a design document.

2

Plan

Breaks work into bite-sized tasks (2-5 minutes each). Every task has exact file paths, complete code, and verification steps. Uses git worktrees for isolated workspaces.

3

Build

Subagent-driven development. Dispatches a fresh subagent per task with two-stage review: first spec compliance, then code quality. Enforces TDD at every step.

4

Review

Automated code review between tasks. Reports issues by severity. Critical issues block progress. The agent checks for relevant skills before every task.

5

Finish

Verifies all tests pass, presents options (merge, PR, keep branch, or discard), and cleans up the worktree. Systematic debugging if anything fails.

Skills Library

Test-Driven Development

RED-GREEN-REFACTOR enforced at every step. Write a failing test first, watch it fail, write minimal code to pass, then refactor. Includes testing anti-patterns reference. If code gets written before tests, the skill deletes it.

Red (fail) Green (pass) Refactor Commit

Systematic Debugging

4-phase root cause process with defense-in-depth and condition-based waiting. No guessing, no random fixes. Includes verification-before-completion so bugs are actually fixed, not just patched.

Isolate Trace root cause Fix Verify

Subagent-Driven Development

Dispatches parallel subagents for concurrent work. Each subagent gets a fresh context, works one task, then gets reviewed in two stages: does it match the spec, then is the code quality good. Fast iteration without context pollution.

Architecture and Business Restructuring

What businesses actually need to ship faster, build better products, and stop wasting time on the wrong things.

Product Strategy

/office-hours + /plan-ceo-review

Before writing a single line of code, challenge every assumption. /office-hours asks the six forcing questions that expose demand reality, your actual wedge, and what the status quo is really costing. /plan-ceo-review finds the 10-star product hiding inside a feature request. Four scope modes let you dream big or strip to essentials.

  • Expose demand reality before building
  • Find the narrowest wedge to ship first
  • Challenge premises and expand scope when it creates a better product
  • Generate implementation alternatives with effort estimates

Technical Architecture

/plan-eng-review

Lock in the execution plan before building. ASCII diagrams for data flow and state machines. Edge cases, error paths, test coverage, performance concerns. Forces hidden assumptions into the open so you don't discover them in production.

  • Data flow diagrams and state machines
  • Edge case identification
  • Test matrix and failure modes
  • Performance and security concerns upfront

Design Systems

/design-consultation + /design-shotgun

Build a complete design system from scratch. Research the competitive landscape, propose creative risks, generate realistic mockups. /design-shotgun shows you 4-6 variants side by side so you pick visually instead of describing in words. Taste memory learns what you like over time.

  • Complete design system creation
  • Competitive landscape research
  • Visual variant comparison
  • Production-ready HTML/CSS output

Developer Experience

/plan-devex-review + /devex-review

If you are building for developers (APIs, SDKs, CLIs), this reviews your onboarding experience. Explores developer personas, benchmarks time-to-hello-world against competitors, and traces friction points. The live audit actually tests your docs and getting started flow.

  • Developer persona exploration
  • Time-to-hello-world benchmarking
  • Friction point mapping
  • Live testing of docs and onboarding

Security Posture

/cso

Infrastructure-first security audit. Secrets archaeology (are there leaked API keys in git history?), dependency supply chain analysis, CI/CD pipeline security, and LLM/AI security. Two modes: daily zero-noise checks and monthly comprehensive deep scans.

  • OWASP Top 10 + STRIDE threat modeling
  • Secrets and credential scanning
  • Dependency supply chain analysis
  • Concrete exploit scenarios for every finding

Shipping and Deployment

/ship + /land-and-deploy + /canary

The full release pipeline. /ship syncs main, runs tests, audits coverage, pushes, and opens a PR. /land-and-deploy merges, waits for CI, and verifies production health. /canary monitors post-deploy for console errors, performance regressions, and page failures.

  • Automated test and coverage audit
  • One-command PR creation
  • Post-merge production verification
  • Continuous canary monitoring

A single person with gstack can do the work of a full engineering team. That means startups ship faster, small businesses get enterprise-quality code, and solo builders stop cutting corners. The sprint workflow ensures nothing gets skipped: every feature gets a product review, architecture review, design review, code review, QA pass, security audit, and proper deployment. That is the standard big companies set with 20-person teams. Now one person can hit it.

Parallel Sprints

gstack is powerful with one sprint. It is transformative with ten running at once.

10-15 Parallel Workers

Using Conductor, you can run multiple Claude Code sessions in parallel, each in its own isolated workspace. One session on /office-hours for a new idea, another doing /review on a PR, a third implementing a feature, a fourth running /qa on staging. All at the same time.

The Sprint Structure Makes It Work

Without a process, ten agents is ten sources of chaos. With a process (Think, Plan, Build, Review, Test, Ship, Reflect), each agent knows exactly what to do and when to stop. You manage them the way a CEO manages a team: check in on the decisions that matter, let the rest run.

100% Test Coverage is the Goal

/ship bootstraps test frameworks from scratch if your project doesn't have one. Every /ship run produces a coverage audit. Every /qa bug fix generates a regression test. Tests make vibe coding safe instead of yolo coding.
